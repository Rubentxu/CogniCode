# DRAFT: Schema/DTO Boundary Enforcement

**Status:** DRAFT — requires human review before promotion to numbered ADR
**Date:** 2026-06-11
**Source:** auto-grill-loop Q001-P1, Q008-P1

## Context
The CogniCode codebase has two overlapping type layers for MCP data transfer:
- `cognicode-core/src/interface/mcp/schemas.rs` — wire-format types
- `cognicode-core/src/application/dto/` — internal transfer objects

These layers have grown independently, with ~5400 lines of near-duplicate types and a `dto_mapping.rs` bridge file that is dead code (zero production callers). Currently, `schemas.rs` directly imports from `application::dto`, breaking Clean Architecture layer discipline.

## Decision
1. **Schemas own the wire format.** DTOs are internal transfer objects. The boundary rule: `schemas.rs` MUST NOT import from `application::dto`.
2. **Newtypes, not aliases.** All 24 mirrored schema/DTO pairs use newtypes generated by a `#[newtype]` attribute macro. No type aliases — 87.5% of pairs differ semantically in their derive/serde contracts.
3. **Per-type derives.** Each newtype explicitly declares its derives. No "standardized" derive set — derives encode semantics, not boilerplate.

## Rationale
- Type aliases silently couple MCP wire format to internal domain types
- Newtypes provide a compilation firewall — MCP evolution doesn't leak into internal consumers
- 87.5% of mirrored pairs already differ in derives, serde attributes, or nested element types
- `#[newtype]` macro eliminates boilerplate while preserving per-type semantic control

## Consequences
- `schemas.rs` removes its `application::dto` import
- `dto_mapping.rs` is deleted (dead code)
- New `#[newtype]` proc macro in `cognicode-macros`
- C4 (Schema/DTO Unification) becomes Wave 3 prerequisite for C1 (Tool Registry)

## Alternatives Considered
- **Type aliases where identical:** rejected — aliases erase the MCP/domain boundary, risking silent coupling
- **Standardized derives:** rejected — derives encode type semantics; blind homogenization is an anti-pattern
- **Separate `cognicode-serde` crate:** rejected — YAGNI; serde is already a workspace dependency

- [ ] Schemas.rs has zero `application::dto` imports
- [ ] `dto_mapping.rs` is deleted with zero build breakage
- [ ] MCP wire format round-trips correctly (serde unchanged)
