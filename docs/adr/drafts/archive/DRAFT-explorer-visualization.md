# DRAFT ADR: Explorer Graph Visualization — Custom SVG + WASM Sugiyama Layout

- **Status:** draft (auto-generated by auto-grill-loop)
- **Date:** 2026-06-07
- **Source:** Q009-P2 (Pass 2)

## Context

The Explorer frontend needs a graph visualization for code structure exploration. The graph is the central feature — users navigate code relationships by clicking nodes to expand the call graph. The `cognicode-diagram` Rust crate already implements a Sugiyama layered graph layout algorithm, producing positioned node coordinates.

Two paths exist:
1. **Graph rendering library** (React Flow, D3, Cytoscape.js) — full graph manipulation with bundled layout engines, interaction models, and rendering.
2. **Custom SVG rendering** — React SVG components positioned by the existing WASM layout engine, with hand-rolled click-to-expand navigation.

The Explorer interaction model is click-to-expand, not graph manipulation. Users do not drag nodes, add edges, or reorganize the layout manually. Every click triggers a WASM layout recomputation and re-render.

## Decision

**Custom React SVG components fed by the WASM Sugiyama layout engine. Zero graph rendering library.**

- **Layout:** Computed in Rust/WASM by `cognicode-diagram` (Sugiyama algorithm). Output: positioned nodes with (x, y) coordinates and edge paths.
- **Rendering:** React SVG components (`<GraphNode>`, `<GraphEdge>`, `<GraphViewport>`) render positioned shapes from layout output.
- **Interaction:** Click on node → WASM recomputes layout → React re-renders SVG. No drag, no force simulation, no node editing.
- **Styling:** SVG reads CSS custom properties via `var(--color-*)` for consistency with the rest of the UI (see ADR design-tokens).
- **Bundle impact:** ~0KB added (rendering code is ~100-200 lines of React SVG) vs React Flow ~200KB.

## Why

1. **Layout already exists.** The hard part (node positioning algorithm) is solved in Rust. Rendering positioned circles and lines as SVG is straightforward React — `<circle cx={x} cy={y} />`, `<line x1={...} y1={...} />`, etc. ~100-200 lines, not a library.
2. **No manipulation features needed.** React Flow, D3, and Cytoscape.js solve graph manipulation (dragging nodes, adding edges, force simulation, zoom-to-fit). None of these are Explorer features. Click-to-expand with WASM recompute is the only navigation model.
3. **Bundle cost is real.** React Flow adds ~200KB gzipped. For features the user will never invoke. This is dead code that slows initial load for every user.
4. **Full control over accessibility.** Custom SVG means we control every ARIA attribute, keyboard handler, and focus behavior — critical for WCAG 2.2 AA compliance (see ADR explorer-accessibility).
5. **Follows minimize_dependencies pattern.** Established by Q003-P1 (component libraries) and Q004-P1 (state management), extended to visualization: only add dependencies for genuinely complex features. SVG rendering of positioned shapes is not complex.

## Consequences

- **Positive:**
  - Zero visualization library bundle cost (~200KB saved vs React Flow)
  - Full control over SVG accessibility (ARIA labels, keyboard navigation to graph data)
  - Tight integration with WASM layout engine — no impedance mismatch between layout library and rendering library
  - Consistent styling via CSS custom properties (graph nodes use same color tokens as UI components)
  - No dependency on a graph library's release cadence, breaking changes, or React version compatibility
- **Risks:**
  - SVG performance may degrade at >500 nodes — DOM node count per SVG element is real. Canvas fallback may be needed for very large graphs.
  - Edge routing (curved paths, orthogonal edges) must be implemented in either Rust layout or React SVG — not handled by a library.
  - Zoom/pan behavior must be hand-rolled (CSS transform, viewBox manipulation, or d3-zoom mini-dependency).
  - No built-in minimap, overview, or graph-in-graph features — all must be implemented if needed.
