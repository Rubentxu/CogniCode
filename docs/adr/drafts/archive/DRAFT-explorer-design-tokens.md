# DRAFT ADR: Explorer Design Tokens — Tailwind 4 @theme + CSS Custom Properties

- **Status:** draft (auto-generated by auto-grill-loop)
- **Date:** 2026-06-07
- **Source:** Q011-P2 (Pass 2)

## Context

The Explorer frontend needs a design token strategy to ensure visual consistency across React components and SVG graph rendering. Design tokens define the atomic visual values (colors, spacing, radii, typography) that all components reference.

Tailwind CSS 4 introduces the `@theme` directive, which generates CSS custom properties from a centralized token definition. This means Tailwind itself is the token source — values defined in `@theme` become both Tailwind utility classes AND CSS custom properties accessible to any element (including SVG).

The alternative is an external token pipeline (Style Dictionary, Tokens Studio, Figma Tokens) that generates platform-specific token files (CSS, JS, SVG). At MVP scale with ~20 semantic tokens and a single platform (web), this pipeline adds tooling maintenance overhead without proportional benefit.

The Explorer launches as dark-only MVP. Light mode is deferred until user demand materializes.

## Decision

**Tailwind CSS 4 `@theme` directive is the single source of truth for all design tokens. No external token pipeline. SVG components read CSS custom properties directly via `var(--color-*)`. Dark-only MVP — light mode deferred.**

Specifics:

| Concern | Strategy |
|---|---|
| **Token source** | Tailwind CSS 4 `@theme` block in the main CSS file. ~20 semantic tokens for colors, spacing, radii, typography scale. |
| **Token consumption (React)** | Tailwind utility classes (`bg-surface`, `text-primary`, `p-4`, `rounded-lg`) |
| **Token consumption (SVG)** | CSS custom properties (`var(--color-surface)`, `var(--color-edge)`, `var(--color-node)`) read directly from the DOM |
| **Token pipeline** | None. No Style Dictionary, Tokens Studio, or Figma Tokens integration at MVP. |
| **Color mode** | Dark-only. `@theme` defines dark values directly. Light mode deferred. |
| **Documentation** | Token values are self-documenting in the `@theme` block. A token catalog page can be added later. |

## Why

1. **Tailwind 4 @theme IS the token system.** It generates CSS custom properties natively. Adding Style Dictionary on top duplicates the same values in a different format — pure overhead at this scale.
2. **20 tokens is manageable inline.** A single `@theme` block with ~20 values (8 colors, 4 spacing steps, 3 radii, 5 font sizes) is readable, reviewable, and diffable. Token pipelines add value at 100+ tokens with multi-platform output — not at MVP scale.
3. **SVG reads CSS custom properties directly.** `var(--color-node)` in an SVG `<circle fill>` stays synchronized with `bg-node` in a React component. No bridge layer, no token export step, no desync risk.
4. **Dark-only defers complexity.** Building a light/dark token system (with semantic aliasing, mode detection, and toggle mechanics) before light mode exists is premature. Dark-only means tokens map 1:1 to values — no indirection.
5. **Reversible decision.** If the project grows to multi-platform (iOS, Android, design tooling), Style Dictionary can be introduced later with the existing `@theme` values as the input source. The migration path is: `@theme` values → feed into Style Dictionary → generate multi-platform output.

## Consequences

- **Positive:**
  - Zero token pipeline maintenance — no Style Dictionary config, no token build step, no platform-specific output files
  - SVG graph styling stays synchronized with React components via shared CSS custom properties
  - Single source of truth in one `@theme` block — grep-able, reviewable in PRs
  - No token synchronization risk between design tools and code (design tools are out of scope for MVP)
- **Risks:**
  - If a second platform (iOS, Android, design tooling) is needed, token extraction becomes necessary. Mitigation: Tailwind `@theme` values are already structured key-value pairs — the input format for Style Dictionary is trivial to derive.
  - If light mode is needed, the `@theme` block must be restructured with semantic aliasing (e.g., `--color-surface: light-dark(var(--color-white), var(--color-gray-900))`). Dark-only defers this complexity.
  - No design tool synchronization — tokens exist only in code. Acceptable for MVP; Figma integration can be added later.
  - Token documentation requires a separate catalog page if developers need a visual reference beyond the `@theme` block.
