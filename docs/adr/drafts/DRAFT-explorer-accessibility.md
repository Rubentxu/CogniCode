# DRAFT ADR: Explorer Accessibility Strategy — Roving Tabindex + axe-core + WCAG 2.2 AA

## Context

The Explorer frontend is a single-page exploration tool with Miller Columns (cascading column file/entity browser), a graph visualization panel, and a command palette (Spotter via cmdk). It has no WAI-ARIA precedent for its most distinctive interaction pattern: cascading-column keyboard navigation with arrow-key traversal across columns.

Accessibility is a first-class requirement, not an afterthought. The component architecture (direct ARIA + Tailwind, see ADR explorer-component-architecture) already establishes that every component owns its accessibility implementation. This ADR defines the cross-cutting accessibility strategy: focus management model, ARIA role assignments, automated enforcement, and compliance target.

## Decision

**Roving tabindex across Miller Columns. WAI-ARIA listbox/option roles for selectable items. Graph as complementary region. axe-core integrated into Playwright CI. WCAG 2.2 AA compliance target. No third-party accessibility library.**

Specifics:

| Concern | Strategy |
|---|---|
| **Focus management** | Roving tabindex across Miller Columns: Tab/Shift+Tab moves between columns; Arrow Up/Down moves within a column; Enter expands selected item; Escape closes current column and returns focus to parent |
| **Column items** | `role="listbox"` on column container, `role="option"` with `aria-selected` on each item |
| **Graph view** | `role="complementary"` with `aria-label` describing graph content; keyboard-accessible data view as fallback for screen readers |
| **Command palette** | Handled by cmdk internally (focus trap, listbox, text input — cmdk is already accessible) |
| **View tabs** | WAI-ARIA Tabs Pattern: `role="tablist"`, `role="tab"`, `aria-selected`, Arrow Left/Right navigation |
| **Automated enforcement** | axe-core integrated into Playwright E2E tests; runs on every CI push; violations block merge |
| **Compliance target** | WCAG 2.2 Level AA |
| **Manual validation** | Screen reader testing (VoiceOver + NVDA) for Miller Columns navigation and graph data access |

## Why

1. **Roving tabindex is the right WAI-ARIA pattern for Miller Columns.** The composite widget with single active item model maps naturally to cascading columns: one focused item at a time, Tab moves between composite widgets (columns), arrows navigate within. No library needed — this is a direct pattern.
2. **listbox/option gives screen readers the right semantics.** Each column is a selectable list. `aria-selected` communicates which item has focus. `aria-label` or `aria-labelledby` on the listbox identifies the column context.
3. **Graph as complementary region is pragmatic.** SVG graphs are inherently visual — screen readers cannot interpret node positions. Providing `aria-label` describing the graph + a keyboard-accessible data table for graph data is the realistic AA path. Don't over-engineer inaccessible visualizations.
4. **axe-core in CI enforces without manual effort.** Automated accessibility checks catch ~30-40% of WCAG violations (color contrast, missing labels, invalid ARIA). The remaining 60% (keyboard navigation, focus order, screen reader UX) requires manual testing — but axe-core eliminates the easy-to-miss mechanical violations.
5. **WCAG 2.2 AA is the industry standard.** It's the legal baseline for public-sector and enterprise software. Meeting AA ensures the Explorer is usable by developers with visual, motor, and cognitive disabilities.
6. **No library — same minimize_dependencies pattern.** WAI-ARIA patterns are direct attributes. Adding react-aria, downshift, or reach-ui would duplicate patterns already documented in APG at ~60 lines each.

## Consequences

- **Positive:**
  - Automated accessibility enforcement in CI prevents regressions
  - Roving tabindex provides consistent, predictable keyboard navigation
  - WCAG 2.2 AA compliance meets enterprise and public-sector requirements
  - Zero accessibility library dependencies
  - Every component tested via React Testing Library's accessibility-role queries (already decided in Q006-P1)
- **Risks:**
  - Miller Columns roving tabindex has NO WAI-ARIA precedent — this is a novel interaction pattern. Must be spiked and validated with real screen reader users (VoiceOver + NVDA) before implementation.
  - SVG graph accessibility is inherently limited — the "complementary region + off-screen data table" pattern may not satisfy all users. Alternative: sonification or textual graph descriptions.
  - axe-core only catches ~30-40% of violations. Manual testing burden remains for keyboard navigation, focus order, and screen reader UX.
  - No third-party component library to lean on for accessibility — every pattern is hand-implemented. Discipline required.
