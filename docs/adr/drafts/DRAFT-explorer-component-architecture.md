# DRAFT ADR: Explorer Component Architecture — Direct ARIA + cmdk

- **Date:** 2026-06-07

## Context

The Explorer frontend (`apps/explorer-ui/`) needs a component architecture strategy. The component tree includes ~8 distinct components: MillerColumn, ViewTabs, ListRow, CardGrid, Playground, ColumnHeader, Breadcrumb, and Spotter (cmd+K overlay).

The default instinct is to adopt a component library (Radix UI, shadcn/ui, etc.) as a project-level dependency. However, per-component analysis reveals that only one component (Spotter) has genuine complexity (portal, focus trap, overlay, text filtering) that warrants a third-party library. The remaining seven components are standard ARIA patterns covered by the WAI-ARIA Authoring Practices Guide at ~60 lines of TypeScript each.

Miller Columns — the most architecturally distinctive component — has no library equivalent regardless, so a custom implementation was always required for its cascading-column keyboard navigation model.

## Decision

**cmdk is the sole third-party component dependency for the Explorer frontend.** All other components use direct ARIA attributes + Tailwind CSS 4 styling with no external component library dependencies.

Specifics:

| Component | Strategy | Notes |
|---|---|---|
| **Spotter** | cmdk | Portal, focus trap, overlay, text filtering — genuine complexity |
| **ViewTabs** | Direct ARIA + Tailwind | WAI-ARIA Tabs Pattern (APG), ~60 lines |
| **Breadcrumb** | Direct ARIA + Tailwind | WAI-ARIA Breadcrumb Pattern (APG) |
| **ListRow** | Direct ARIA + Tailwind | Listbox pattern with `role="option"` |
| **CardGrid** | Direct ARIA + Tailwind | Grid pattern with `role="grid"` |
| **Playground** | Direct ARIA + Tailwind | Application region with live region updates |
| **ColumnHeader** | Direct ARIA + Tailwind | Column header with sort controls |
| **MillerColumn** | Custom keyboard nav + Tailwind | Cascading-column focus model — no library helps |

- **No direct Radix UI dependencies.** cmdk vendors its own Radix Dialog internally.
- **No shadcn/ui.** Its opinionated wrapper layer adds abstraction without benefit for this component set.
- **Tailwind CSS 4** for all styling. CSS variables from the prototype map into Tailwind `@theme` namespaces.

## Why

1. **Per-component evaluation beats per-project defaults.** Only 1 of 8 components has complexity that justifies a library. Adding a full component library for the other 7 is dead weight.
2. **Miller Columns are custom regardless.** The hardest component was always going to be hand-built — no Radix, Base UI, or shadcn/ui primitive covers cascading-column keyboard navigation with arrow-key traversal.
3. **WAI-ARIA APG covers the rest.** Tabs, Breadcrumb, Listbox, Grid patterns are well-documented with reference implementations. Each is ~60 lines of TypeScript with `role`, `aria-*`, and keyboard handlers.
4. **cmdk is proven for command palettes.** Its focus trap, portal, text filtering, and overlay behavior are production-tested. Reimplementing Spotter from scratch would be the real waste.
5. **Small dependency surface.** One dependency (cmdk) is auditable, upgradeable, and replaceable. A component library like Radix creates a transitive dependency on its entire maintenance trajectory.

## Consequences

- **Positive:**
  - Minimal dependency surface (1 library, not 1 library suite)
  - Full control over accessibility implementation per component
  - No lock-in to a component library's release cadence or breaking changes
  - Smaller bundle — no unused primitives shipped
  - Each component is independently testable with standard testing-library/react + jest-dom ARIA assertions
- **Risks:**
  - Miller Columns keyboard navigation has no WAI-ARIA precedent — needs a spike and user validation before implementation
  - cmdk's vendored Radix Dialog may have React 19 compatibility issues (must verify)
  - Direct ARIA requires discipline — each component must be tested against the corresponding WAI-ARIA pattern
  - Future components with genuine complexity may require revisiting this decision (acceptable — evaluate per component, not per project)
