## Context

The Explorer frontend needs a technology stack. The workspace already contains a Leptos (Rust WASM) dashboard with 18 pages, 18 components, and 61 Playwright tests. ADR 0009 previously selected React 19 + Tailwind CSS for web frontends. This created a potential architectural fork: should the Explorer extend the existing Leptos dashboard, or use the React stack specified in ADR 0009?

The auto-grill-loop initially recommended extending Leptos to avoid a technology fork. The user overrode this recommendation.

## Decision

**React 19 + TypeScript strict mode is the technology stack for the Explorer frontend.**

Specifics:
- TypeScript with `strict: true` (strictNullChecks, noImplicitAny, etc.)
- `.tsx` for components, `.ts` for utilities and logic
- Domain types generated from Rust DTOs via `ts-rs` into shared `@cognicode/types` package
- Clean rewrite from the 478-line vanilla JS prototype (prototype is spec only)
- The existing Leptos dashboard is a **separate, independent application** — no code sharing, no technology coupling between Dashboard and Explorer

## Why

The user explicitly stated the dashboard is an independent application. Explorer and Dashboard serve different purposes and have different technology needs. ADR 0009 already established React 19 + Tailwind CSS as the stack for web frontends. This decision reaffirms and applies that ADR to the Explorer specifically.

## Consequences

- **Positive:**
  - Clear technology boundary between two independent products
  - TypeScript strict mode provides type safety across the Explorer codebase
  - Follows existing ADR 0009 — no new technology introduced
- **Risks:**
  - Two frontend stacks to maintain (React + Leptos)
  - ts-rs fragility: community crate with limited maintainers — complex Rust enum codegen may break
  - Build pipeline complexity: cargo build → ts-rs codegen → npm install → tsc → vite build
  - Any shared concerns (auth, API contracts) need deliberate decoupling design
