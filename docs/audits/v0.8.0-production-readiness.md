**❌ NO production-ready.** La arquitectura está bien diseñada pero la **integración está incompleta**. Funciones críticas existen pero no están conectadas al flujo del usuario. El cambio se cerró prematuramente sin tests de integración end-to-end del flujo completo de Exploration Snapshot.

**Grado:** **C+** (bug fix core funciona, persistencia UI no integrada)

## 2. Hallazgos Críticos (🔴 BLOQUEAN PRODUCCIÓN)

### 🔴 H1: `useSnapshotCache` definido pero NO se usa

**Archivo:** `apps/explorer-ui/src/hooks/useExplorations.ts:45-79`

**Problema:** El hook `useSnapshotCache` está exportado pero NO se importa ni se llama desde ningún componente. El localStorage cache **nunca se escribe**.

**Evidencia:**
```bash
$ grep -r "useSnapshotCache" apps/explorer-ui/src/
apps/explorer-ui/src/hooks/useExplorations.ts:45: export function useSnapshotCache(
# solo en la definición
```

**Impacto:** Los usuarios NO tienen cache local. Recargar la página = perder todo el estado de exploración. La feature de persistence híbrida está **rota**.

**Fix:** Llamar `useSnapshotCache(workspaceId, sessionId, panes)` desde `Shell.tsx` o `PaneStackView.tsx`.

---

### 🔴 H2: `saveExplorationSession` definido pero NO se usa

**Archivo:** `apps/explorer-ui/src/hooks/useExplorations.ts:121-160`

**Problema:** La función existe y es correcta, pero NO se llama desde `ShareExplorationButton.tsx`. El botón usa `fetch()` directo sin validación Zod ni manejo de errores.

**Evidencia:**
```typescript
// ShareExplorationButton.tsx:29-41 (raw fetch)
const resp = await fetch("/api/exploration-sessions", {
  method: "POST",
  headers: { "Content-Type": "application-Type" },
  body: JSON.stringify({
    workspace_id: "default",
    events,
    navigation_mode: "pane-stack",
    // ⚠️ NO INCLUYE `panes` ni `viewport`!
  }),
});
```

**Impacto:** El Share button **no guarda el viewport state** aunque el backend lo soporta. La persistencia es parcial.

**Fix:** Reemplazar el `fetch()` directo con `await saveExplorationSession(workspaceId, events, panes)`.

---

### 🔴 H3: `useRestoreExploration` ignora `panes` field del response

**Archivo:** `apps/explorer-ui/src/hooks/useRestoreExploration.ts:23-37`

**Problema:** Cuando el usuario abre un link con `?exploration=<id>`, el hook restaura solo `events` como `PUSH_PANE` pero **ignora el campo `panes` con viewport state**.

**Evidencia:**
```typescript
.then((session) => {
  for (const ev of session.events) {
    dispatch({ type: "PUSH_PANE", payload: { objectId: ev.object_id, viewId: ev.view_id ?? undefined } });
  }
  // ⚠️ NO restaura viewport, scrollY, ni dedup por paneId
})
```

**Impacto:** Share URLs no restauran el estado visual completo (zoom, pan). El usuario pierde contexto.

**Fix:** Usar `session.panes` para restaurar viewport por paneId.

---

### 🔴 H4: MSW mock no tiene handler para `/api/exploration-sessions`

**Archivo:** `apps/explorer-ui/src/mocks/handlers.ts`

**Problema:** El frontend llama a `POST /api/exploration-sessions` y `GET /api/exploration-sessions/:id`, pero el MSW handler **NO existe**. Los tests e2e con MSW fallan al restaurar exploraciones.

**Evidencia:**
```bash
$ grep "exploration-sessions" apps/explorer-ui/src/mocks/handlers.ts
# 0 matches
```

**Impacto:** Los tests e2e que comparten exploraciones (incluido `exploration-snapshot.spec.ts`) no pueden validar persistencia real. Son tests **triviales** que solo verifican que se abre el shell.

**Fix:** Agregar handlers:
```typescript
http.post("/api/exploration-sessions", ...),
http.get("/api/exploration-sessions/:id", ...),
```

## 3. Hallazgos de Alta Prioridad (🟡)

### 🟡 H5: Pane-Stack solo abre desde graph click — no desde listas/breadcrumbs

**Archivo:** `apps/explorer-ui/src/components/ObjectInspector/PaneInspector.tsx:226-234`

**Problema:** Cuando el usuario hace click en un item de `CallListView` (callers/callees), `onSelectObject` siempre dispatcha `viewId: "overview"`:
```typescript
onSelectObject={(objId) =>
  dispatch({
    type: "SELECT_OBJECT",
    payload: { objectId: objId, viewId: "overview" },
  })
}
```

Pero `GraphViewRenderer` preserva `view.view_id` para mantener el contexto. **Inconsistencia**: drill-down desde graph = "call_graph", drill-down desde list = "overview".

**Impacto:** UX inconsistente. El usuario pierde el tipo de vista al navegar desde lists.

**Fix:** Usar `view.view_id` también en `Blocks onSelectObject` callback.

---

### 🟡 H6: `useEffect` dependency faltante en `PaneInspector`

**Archivo:** `apps/explorer-ui/src/components/ObjectInspector/PaneInspector.tsx:101-111`

**Problema:** El effect de fallback depende de `[views, viewId, objectId, dispatch]` pero `views` puede cambiar frecuentemente sin que la lógica detecte cambios válidos.

**Evidencia:** Si `viewId` cambia pero `views.some((v) => v.id === viewId)` retorna `true`, el effect sale temprano — comportamiento correcto. Pero si `views` cambia por SWR revalidation mientras `viewId` ya no está, puede causar race condition.

**Impacto:** UX intermitente: occasionalmente abre un pane incorrecto en navigation rápida.

**Fix:** Considerar usar `useRef` para comparar `views` o memoización con selector.

---

### 🟡 H7: `state.context.ts` reducer no maneja `UPDATE_PANE_VIEWPORT`

**Archivo:** `apps/explorer-ui/src/state/context.ts` (líneas 80-110)

**Problema:** El reducer principal `appReducer` no tiene case para `UPDATE_PANE_VIEWPORT`. La acción existe en `types.ts` y se dispatcha en `paneStack.ts`, pero el reducer de nivel superior no la enruta al navigation slice.

**Evidencia:**
```typescript
switch (action.type) {
  case "SET_WORKSPACE": ...
  case "TOGGLE_SPOTTER": ...
  case "SET_SPOTTER": ...
  case "ADD_EXPLORATION": ...
  case "RESET": ...
  case "SET_PERSPECTIVE": ...
  default: { /* exhaustive check */ }
}
```

**Impacto:** El viewport state **nunca se persiste** en el state global. `useSnapshotCache` recibe `panes` que NO tienen `viewport` actualizado.

**Fix:** Agregar case que delega a `apply(state.navigation, action)` del paneStack reducer.

---

### 🟡 H8: `MAX_PANES = 8` puede ser insuficiente para comparaciones reales

**Archivo:** `apps/explorer-ui/src/state/navigation/paneStack.ts:26`

**Problema:** Comparar 8 funciones simultáneamente es útil, pero el proyecto plantea "side-by-side comparisons" (comparar implementaciones alternativas). El cap es arbitrario.

**Impacto:** Limitación artificial del moldable flow. GToolkit no tiene cap.

**Fix:** Aumentar a 12-16 o eliminar el cap con virtual scrolling.

## 4. Hallazgos de Prioridad Media (🟢 Polish)

### 🟢 H9: Edge labels solo se ocultan cuando highlighted=false, pero SvgGraph.tsx:301 cambia

**Archivo:** `apps/explorer-ui/src/components/SvgGraph/SvgGraph.tsx:286-305`

**Problema:** El fix "labels only when highlighted" se aplica, pero `GraphEdge.tsx` puede renderizar el label siempre que se pase. La condición `highlighted ? edge.label : undefined` está en SvgGraph, lo cual es correcto, pero el `data-testid` no incluye el label lo que dificulta debugging.

---

### 🟢 H10: Falta confirmación visual al guardar exploration

**Archivo:** `apps/explorer-ui/src/components/ShareExplorationButton.tsx:46-50`

**Problema:** El botón muestra "✓ Copied!" pero no confirma que el servidor recibió la sesión. Si el POST falla, el usuario copia un URL inválido.

**Fix:** Mostrar "Saving..." mientras se hace POST, "Saved" o "Failed" después.

---

### 🟢 H11: `useSnapshotCache` quota exceeded silenciado

**Archivo:** `apps/explorer-ui/src/hooks/useExplorations.ts:71-73`

**Problema:** Si localStorage excede quota (~5MB), el `catch` silencioso hace que el usuario pierda datos sin saberlo.

**Fix:** Mostrar toast warning "Snapshot cache full — older snapshots deleted".

---

### 🟢 H12: No hay loading state en `GraphViewRenderer`

**Archivo:** `apps/explorer-ui/src/components/GraphView/GraphViewRenderer.tsx`

**Problema:** No hay skeleton/spinner cuando `layoutFromContextualView` está calculando (puede ser lento con muchos nodos).

**Fix:** Agregar `<GraphLoadingSkeleton />` durante el cálculo.

## 8. Resumen

**El cambio v0.8.0 corrige el bug crítico del Call Graph correctamente y entrega la arquitectura Moldable Navigation. Sin embargo, la integración end-to-end de Exploration Snapshot tiene 4 gaps bloqueantes (H1-H4) que impiden que el feature funcione como fue diseñado.**

**Recomendación:** NO mergear a producción hasta resolver H1-H4. Esos son ~3 horas de trabajo que completan la integración.

**Calificación final:** **C+** — buena arquitectura, integración incompleta.
