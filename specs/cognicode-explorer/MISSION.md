# Mission: Visual Software Understanding Workflow

## Why
Build a practical way for developers to understand how an application is composed, navigate its structure visually, and make better design decisions using knowledge graphs, diagram DSLs, moldable development, visual thinking, and software design heuristics.

## Success looks like
- Explain the difference between a code graph, an architecture diagram, and a moldable view.
- Use Graphify output as evidence for developer-facing visualizations.
- Choose the right diagram DSL for a specific architectural question.
- Evaluate software structure using design concepts such as complexity, module depth, information hiding, and design alternatives.
- Sketch a workflow for an application that turns repository evidence into useful visual decision views.

## Constraints
- Keep learning tied to a future visualizer/workflow application, not abstract diagramming theory.
- Prefer trusted sources and observable repository evidence over intuition.
- Teach in small steps with exercises that produce concrete artifacts.

## Out of scope
- Building the full application before the conceptual workflow is clear.
- Treating diagrams as decoration rather than decision tools.
